How the QA Workspace Works

Inputs, workflow, parallel agents, scoring, artifacts, and report pipeline — everything you need to understand before starting an audit.

Section 01

What You Need to Provide

Every audit requires exactly two inputs. Without both, the audit cannot start.

Required — both must be provided before any audit begins:
1. A GitHub repository URL (for source code analysis, dependency checks, CI/CD review, secrets scanning)
2. A live site URL (for HTTP checks, browser testing, performance metrics, accessibility scanning)
GitHub Repo URL
Source Code Access
Used for: code quality, dependency health, secrets detection, CI/CD config, compliance patterns, framework-specific checks.
Live Site URL
Runtime Access
Used for: HTTP headers, TTFB, accessibility scanning, SEO meta, broken links, browser flows, Core Web Vitals.
Optional: Auth Credentials
Login-Gated Apps
If the app has a login wall: provide API token / email+password, or log in via Brave browser (CDP session is reused).
Optional: Risk Profile
Adjusts Score Weights
High data sensitivity → Security weight increases to 30%. E-commerce → Compliance weight increases. Default weights apply otherwise.

Section 02

Main Audit Workflow

The full audit runs in 7 phases. Phases 3 and 4 execute simultaneously across 10 background agents.

Section 03

The 10 Parallel Audit Agents

Each agent is a background Task specialized for one audit area. All 10 run simultaneously.

Agent ID Prefix Primary Tool What It Checks Score Weight
Security S-xx Python + httpx HTTP headers, HTTPS, CORS, OWASP checks, auth review, input validation, API surface 25%
Accessibility A-xx Python + BeautifulSoup4 WCAG 2.2 AA: labels, ARIA roles, landmarks, headings hierarchy, form accessibility, color contrast 20%
Performance P-xx Python + httpx Core Web Vitals, TTFB, page weight, CLS, resource optimization, caching headers 15%
SEO E-xx Python + BeautifulSoup4 Meta tags, Open Graph, robots.txt, sitemap, structured data, canonical URLs, heading structure 10%
Code Quality C-xx Grep + Read Hardcoded secrets, type safety, architecture patterns, dead code, naming conventions 15%
Functional FN-xx agent-browser (Brave CDP) Navigation flows, form submission, edge cases, error handling, resilience, broken links 15%
Dependencies D-xx Python scripts + npm audit Vulnerable packages, outdated deps, license compatibility, supply chain risks —
Compliance R-xx Grep + Read + httpx GDPR, CCPA, cookie consent banner, privacy policy, Terms of Service coverage —
Operations O-xx Grep + Read CI/CD pipelines, monitoring setup, secrets management, disaster recovery, rollback strategy —
Framework FW-xx Grep + Read Astro islands / Angular change detection, SSR config, bundle optimization, framework best practices —

Score weights apply to the weighted overall score. Security, A11y, Perf, Code, and Func are the primary weighted dimensions. Deps, Compliance, Ops, and Framework scores are reported individually and influence the verdict qualitatively.

For login-gated applications, the agent follows a three-tier approach, preferring automated token auth.

Section 05

Scoring Formula & Launch Verdict

Category Score (per audit area)
Score = 10 − (Critical × 2.0 + High × 1.0 + Medium × 0.4 + Low × 0.1)
Clamped to [0, 10]
Overall Score (weighted average)
Overall = Security×25% + A11y×20% + Perf×15% + Code×15% + Func×15% + SEO×10%
Weights adjust based on risk profile (e.g., high data sensitivity → Security increases to 30%)
Verdict A
READY
Overall ≥ 8.0
Zero Critical findings
Zero unresolved High findings
Verdict B
READY WITH RESERVES
Overall 6.0 – 7.9
Zero Critical findings
High findings have a documented remediation timeline
Verdict C
NOT READY
Overall < 6.0
— OR —
Any unresolved Critical finding

Section 06

Folder Structure

The QA workspace is a fixed sandbox. Target repos are cloned in, never committed out.

QA/ ← this workspace (git repo: kodexArg/QA) ├── AGENTS.md Agent role, workflow strategy, tool reference ├── .claude/ │ ├── CLAUDE.md QA identity, rules, report format, skills list │ ├── settings.json PostToolUse hook → auto-commit reports │ └── skills/ 11 QA-specific skills (qa-intake, qa-security, etc.) ├── github-repositories/ ← gitignored entirely — never committed │ └── RRHH/ Example clone of luinloder/RRHH (testing sandbox) ├── reports/ ← separate git repo: kodexArg/qa-reports (public) │ ├── template.html Base HTML template for all audit reports │ └── YYYY-MM-DD_owner-repo_type.html Generated reports (auto-pushed) ├── scripts/ Reusable Python automation scripts (uv run) └── context/ Reference PDFs, guides, client briefs
Critical rule: github-repositories/ is the testing sandbox — repos cloned here are never committed or pushed from this workspace. The .gitignore excludes the entire directory, so no per-repo entries are needed. Always git pull origin master inside a clone before starting a new test run.

Section 07

Report Pipeline — Auto-Publish

Reports are written locally and automatically committed and pushed to the public kodexArg/qa-reports GitHub repo via a PostToolUse hook.

sequenceDiagram participant Agent as QA Agent participant Local as reports/ (local git repo) participant Hook as PostToolUse Hook participant GitHub as kodexArg/qa-reports (public) Agent->>Local: Write HTML report\n(YYYY-MM-DD_owner-repo_type.html) Agent->>Local: Write Markdown companion\n(same filename, .md extension) Local-->>Hook: Triggered on every Write/Edit\ninside reports/ Hook->>Local: git add *.html Hook->>Local: git commit -m "auto: add report" Hook->>GitHub: git push origin master GitHub-->>Agent: ✓ Report live at\ngithub.com/kodexArg/qa-reports
Report naming convention: reports/YYYY-MM-DD_github-owner-repo_audit-type.html
Example: 2026-03-01_luinloder-RRHH_full-audit.html
Only .html files are tracked in the qa-reports repo (.gitignore excludes everything else including .md).

Section 08

Browser Testing — Brave + agent-browser

All interactive browser tests use agent-browser connected to Brave via Chrome DevTools Protocol (CDP). This preserves your login session.

flowchart LR U([You launch Brave with CDP]) --> B["brave-browser\n--remote-debugging-port=9222\n--user-data-dir=$HOME/.config/Brave"] B --> V["curl http://localhost:9222/json/version\n→ verify CDP is live"] V --> C["Agent connects:\nnpx agent-browser --cdp 9222"] C --> D1["Open URL:\nnpx agent-browser --cdp 9222 open https://site.com"] C --> D2["DOM snapshot:\nnpx agent-browser --cdp 9222 snapshot -i"] C --> D3["Screenshot:\nnpx agent-browser --cdp 9222 screenshot /path/out.png"] C --> D4["Click / fill forms:\nnpx agent-browser --cdp 9222 click 'button'"] style U fill:#fafaf8,stroke:#1a1a1a style B fill:#eef6fa,stroke:#2e86ab
Do NOT use: the chrome-devtools skill, scoutqa-test skill, or any other browser automation path. Always use agent-browser with Brave CDP only.

Section 09

Quick Reference

To start a full audit
Provide GitHub URL + Live URL. Say "run full audit". The agent runs intake, clones repo, spawns 10 agents in parallel, then writes the report.
To run a single check
Say "run security scan on X" or "check accessibility of X". The agent runs only that area and produces a focused report.
Auth-gated apps
Either provide API credentials (preferred) or log in via Brave first, then ask the agent to connect via CDP port 9222.
Report location
reports/YYYY-MM-DD_owner-repo_type.html
Auto-pushed to github.com/kodexArg/qa-reports
Cloned repos
github-repositories/repo-name/ — entirely gitignored. Pull latest before each run.
Python scripts
Stored in scripts/. Run with uv run. Inline dependencies, no requirements.txt needed.
Finding IDs
S-xx Security · A-xx A11y · P-xx Perf · E-xx SEO · C-xx Code · FN-xx Functional · D-xx Deps · R-xx Compliance · O-xx Ops · FW-xx Framework
Language rule
All outputs (reports, code comments, conversation) are in English regardless of input language. Non-negotiable.

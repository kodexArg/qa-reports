Note: Live URL returns HTTP 403 with x-vercel-mitigated: challenge (bot protection). TTFB and response time measurements reflect the challenge page, not the actual application. HTTP-level findings below are based on the observed headers.

P-01 Medium No Response Compression Detected HTTP Response Headers

Impact: HTML responses are sent uncompressed, increasing transfer size by ~56%. The 31 KB HTML response would be ~13.4 KB with gzip.

Evidence: Content-Encoding: none in all sampled responses. Uncompressed body: 31,056 bytes. Estimated gzipped: 13,700 bytes (44.1% compression ratio).

Remediation:

  • Vercel typically handles compression automatically. Verify Vercel deployment settings.
  • The 403 challenge page may bypass normal compression. Re-test after resolving bot protection.
  • Consider adding compressHTML: true to astro.config.mjs for Astro-level HTML minification.
P-02 Medium Aggressive No-Cache Headers HTTP Response Headers

Impact: Cache-Control: private, no-store, max-age=0 prevents all caching, including browser back/forward cache. Every navigation triggers a full server round-trip.

Evidence: No ETag, no Last-Modified, no Expires header present. Combined with SSR mode, every page request hits the Vercel serverless function and Supabase.

Remediation:

  • For authenticated SSR pages, private, no-store is intentional and correct for security.
  • Consider adding Cache-Control: public, max-age=31536000, immutable for static assets (JS, CSS, fonts, images) via vercel.json headers config.
  • Consider stale-while-revalidate for non-sensitive pages like the dashboard.
P-03 Low Missing HSTS Header HTTP Response Headers

Impact: Without Strict-Transport-Security, initial connections may attempt HTTP before being redirected, adding latency. Also a security concern.

Evidence: strict-transport-security: not set in response headers.

Remediation: Add via vercel.json: "Strict-Transport-Security": "max-age=63072000; includeSubDomains; preload"

Impact: Loading 3 font families (DM Serif Display, DM Sans with 7 weights + italic, JetBrains Mono with 2 weights) downloads ~378 KB of font data across 28 WOFF2 files. Includes unused subsets: cyrillic, cyrillic-ext, greek, vietnamese. This is the single largest contributor to page weight.

Evidence:

fonts.googleapis.com/css2?family=DM+Serif+Display:ital@0;1

Subsets loaded: cyrillic, cyrillic-ext, greek, latin, latin-ext, vietnamese
Font files referenced: 28
Estimated total download: ~378 KB

Remediation:

  • Self-host fonts with only latin and latin-ext subsets. This alone would cut font size by ~60%.
  • Remove DM Sans weight 300 (light) and italic if unused. Check actual usage in CSS.
  • Use font-display: swap (already set via &display=swap parameter -- good).
  • Consider @fontsource packages for self-hosting: @fontsource/dm-sans, @fontsource/dm-serif-display, @fontsource/jetbrains-mono.
  • Alternatively, use astro-font integration for automatic self-hosting and subsetting.
P-05 Medium Large Inline Style Blocks (219 KB total across 55 components) Multiple .astro files

Impact: 219 KB of inline CSS across 55 <style> blocks is sent as part of the HTML on every page load. While Astro scopes these, they cannot be cached independently and inflate every HTML response.

Evidence: Top inline style offenders:

18,021 bytes  pages/residencias/simulador.astro
11,433 bytes  components/cpo/ResumenMesDemanda.astro
11,320 bytes  components/cpo/CalendarioDDJJ.astro
10,007 bytes  components/cpo/PanelPrioridadPromocion.astro

Remediation:

  • Convert repeated patterns to Tailwind utility classes. Many inline styles duplicate Tailwind-available properties.
  • Extract shared component styles (like calendar grid styles) into globals.css @layer components.
  • The simulador page alone has 18 KB of inline styles -- consider refactoring to a dedicated CSS file.
P-06 Low 23 Inline SVG Icons in BaseLayout Navigation src/components/layout/BaseLayout.astro:72-207

Impact: 23 inline SVG icons are embedded directly in the layout HTML, adding ~5 KB to every page load. These cannot be cached separately.

Evidence: 23 <svg> elements found in the sidebar navigation. Each page load repeats all 23 SVGs.

Remediation:

  • Use an SVG sprite sheet (public/icons.svg) with <use href> references.
  • Or use Astro Icon integration (astro-icon) for automatic SVG handling.
  • Low priority since the total impact is small (~5 KB) and gzip handles repetitive SVG well.

03 / 05

Build Configuration

P-07 Medium Full SSR Mode Without Hybrid Optimization astro.config.mjs:8

Impact: output: 'server' renders every page on the server for every request. Pages like /legal/privacidad, /404, and login are re-rendered even though they have no dynamic data. Each request triggers a Vercel serverless function cold start + Supabase auth check via middleware.

Evidence:

// astro.config.mjs
output: 'server',
adapter: vercel(),

The middleware runs auth checks on every request including static-content pages. Combined with sequential Supabase queries (getUser + fetch usuario), each page load incurs at minimum 2 database round-trips.

Remediation:

  • Switch to output: 'hybrid' and mark truly static pages with export const prerender = true (404, legal/privacidad, login).
  • This eliminates serverless function invocations for static pages entirely.
  • No Astro prefetch strategy is configured. Add prefetch: { prefetchAll: true } or prefetch: { defaultStrategy: 'hover' } to astro.config.mjs.
P-08 High Sequential Supabase Queries in Middleware (Waterfall) src/middleware.ts:56-73

Impact: Every authenticated page request executes 2-3 sequential Supabase queries in the middleware: (1) getUser(), (2) fetch usuarios by auth_id, (3) optional fallback fetch by email. This creates a request waterfall adding 100-300ms to TTFB. Additionally, some pages run their own sequential queries.

Evidence:

// middleware.ts - sequential waterfall
const { data: { user } } = await supabase.auth.getUser();  // Query 1
let { data: usuario } = await supabaseAdmin               // Query 2
  .from('usuarios').select(...).eq('auth_id', user.id).single();
// Fallback Query 3 if not found
if (!usuario && user.email) {
  const { data: byEmail } = await supabaseAdmin
    .from('usuarios').select(...).eq('email', user.email)...
}

Pages with additional sequential queries (no Promise.all):

3 sequential queries - pages/informes/[id].astro
2 sequential queries - pages/informes/index.astro
3 sequential queries - pages/informes/[id]/editar.astro
2 sequential queries - pages/residencias/simulador.astro

Remediation:

  • The middleware waterfall is unavoidable (auth_id depends on getUser result), but pages should use Promise.all() for their own queries.
  • Wrap parallel-safe queries in simulador.astro (hospitals + residents fetch) with Promise.all().
  • The dashboard (index.astro) already uses Promise.all -- follow this pattern everywhere.
  • Consider a Supabase edge function or view to combine middleware queries into a single RPC call.

Estimation Methodology

LCP (~1.5s): SSR HTML arrives in ~80ms TTFB. LCP element is likely the hero text or KPI cards (text-based, no large images). However, Google Fonts must load before text renders with correct typeface. With font-display: swap, initial text appears fast with system fonts, then swaps. The main LCP risk is the SSR waterfall (middleware auth + page queries + render).

INP (~50ms): Alpine.js is lightweight (~15 KB) and handles interactions. No heavy client-side frameworks. The simulador page has complex x-data computations but these are local calculations, not DOM-heavy operations.

CLS (~0.02): Minimal CLS risk. SSR delivers full HTML with styles. font-display: swap could cause minor text reflow but the impact is small. No lazy-loaded images or dynamic content insertion patterns detected.

Observed

Strengths

Chart.js dynamically imported via chart-loader.ts with lazy singleton pattern. Only pages with charts load the ~70 KB library.
Astro SSR architecture sends zero JavaScript by default (islands architecture). Alpine.js is the only client-side framework, at just ~15 KB.
No images in page content -- the app uses inline SVG icons and CSS-only visual elements, eliminating image optimization concerns entirely.
Google Fonts preconnect correctly configured with both fonts.googleapis.com and fonts.gstatic.com (with crossorigin).
Dashboard uses Promise.all for parallel Supabase queries (3 queries in parallel on index.astro).
Minimal client-side dependencies -- only Alpine.js and Chart.js ship to client. Supabase, Zod, clsx, and tailwind-merge are server-only.
Rate limiting in middleware protects against abuse without adding external dependencies (in-memory sliding window).
Fire-and-forget heartbeat -- the ultima_actividad update in middleware uses .then() without awaiting, not blocking the response.
Lightweight CSS architecture -- globals.css is only 14 KB pre-purge with well-organized Tailwind layers (base, components, utilities).
font-display: swap configured via Google Fonts URL parameter, preventing font-blocking render.
